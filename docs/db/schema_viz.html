AI 매칭 실행 흐름
1
매칭 대상 수집
applications WHERE application_status = approved
멘티 신청 목록 + 멘토 신청 목록 JOIN해서 수집
각자의 mentee_records / mentor_records 데이터 포함
2
재실행 시 기존 결과 취소
기존 match_results WHERE process_year = 2025
AND match_status != cancelled → cancelled로 UPDATE
Partial Unique Index 덕분에 취소 후 재생성 가능
3
LLM API 호출 → 결과 저장
멘티·멘토 정보를 LLM에 전달
응답으로 ai_score, ai_reason 수신
match_results INSERT
match_status = suggested, is_finalized = false
4
수동 수정 → 임시저장
특정 매칭의 멘토 변경: mentor_application_id UPDATE
match_status = draft, ai_score = null (수동 수정 표시)
임시저장: is_finalized = false 유지
5
매칭 확정
전체 match_status = finalized
is_finalized = true 일괄 UPDATE
멘티·멘토에게 알림 발송
확정 후 수정 시도 → API에서 에러 반환
LLM 호출은 시간이 오래 걸려요. API가 즉시 응답하지 말고 비동기로 처리 후 완료 알림을 보내는 구조를 추천해요.
회원 탈퇴 (Soft Delete) 흐름
1
탈퇴 요청 접수
users UPDATE 실행
is_deleted = true, deleted_at = now()
2
개인 식별 정보 익명화
name = "탈퇴한 사용자"
phone = null, student_id = null
email = "deleted_{user_id}@removed.com"
account_status = inactive
3
신청·매칭 이력 보존
mentee_records, applications, match_results 삭제 안 함
users의 개인정보만 지워지므로 사실상 식별 불가
매칭 품질 개선용 통계 데이터는 유지됨
4
조회 시 필터링
모든 API에서 WHERE is_deleted = false 조건 추가 필요
Prisma middleware로 전역 적용 권장
모든 조회 API에 is_deleted = false 필터를 빠뜨리면 탈퇴 회원이 노출돼요. Prisma의 $extends middleware로 전역 처리하면 실수를 방지할 수 있어요.
Prisma 자동 UNIQUE (user_id, process_year) — mentee_records / mentor_records
같은 사람이 같은 연도에 멘티 레코드를 두 개 만들 수 없어요.
예) 2025년에 홍길동의 멘티 레코드가 이미 있으면 다시 INSERT 시 DB 에러 발생.
prisma migrate dev 실행 시 자동 생성됩니다.
Prisma 자동 UNIQUE (user_id, process_year, role) — applications
같은 사람이 같은 연도에 같은 역할로 신청을 두 개 만들 수 없어요.
단, 같은 사람이 같은 연도에 멘티 + 멘토로 각각 신청하는 건 허용돼요 (role이 다르므로).
prisma migrate dev 실행 시 자동 생성됩니다.
SQL 수동 추가 CHECK (role별 record_id 무결성) — applications
role = 'mentee' → mentee_record_id IS NOT NULL AND mentor_record_id IS NULL
role = 'mentor' → mentor_record_id IS NOT NULL AND mentee_record_id IS NULL

Prisma가 자동 생성하지 않으므로 migration.sql 맨 아래에 직접 추가해야 해요:
ALTER TABLE applications ADD CONSTRAINT check_record_by_role CHECK (...);
SQL 수동 추가 Partial Unique Index — match_results
취소(cancelled)되지 않은 매칭 중 같은 멘티는 하나의 매칭만 가질 수 있어요.
AI 매칭을 재실행하면 기존 결과를 cancelled 처리하고 새로 생성하는 방식이 가능해지는 근거예요.

CREATE UNIQUE INDEX uq_active_mentee_match ON match_results (mentee_application_id) WHERE match_status != 'cancelled';

PostgreSQL 전용 기능이라 Prisma @@unique로는 불가능해요.
API 레벨 매칭 확정 후 수정 불가
is_finalized = true인 매칭에 대한 수정 요청 시 API에서 400 에러를 반환해야 해요.
DB 제약조건이 아니라 API 로직에서 체크하는 부분이에요.
if (match.is_finalized) throw new Error('확정된 매칭은 수정할 수 없습니다')
API 레벨 탈퇴 회원 전역 필터링 (is_deleted)
모든 조회 쿼리에 where: { is_deleted: false }를 넣어야 해요.
Prisma middleware로 전역 적용하면 실수를 방지할 수 있어요:
prisma.$extends({ query: { user: { findMany: ({ args, query }) => query({ ...args, where: { ...args.where, is_deleted: false } }) } } })
